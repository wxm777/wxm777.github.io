HawkEye-Scanner：纯离线 JWT 弱密钥高并发探测器

发表于 2026-05-22 分类于 安全研发 (SecDev)

在微服务大行其道的今天，JWT (RFC 7519) 已经成为 API 鉴权的基础设施。然而，在针对大规模资产进行鉴权基线审计时，如何构建一个既具备深度探测能力，又能在脏数据中稳定存活的扫描工具，始终是安全研发的一道工程考题。

本文将复盘 HawkEye-Scanner 的核心设计逻辑，探讨如何摒弃臃肿的第三方框架，通过对密码学原理的拆解与底层字节流的严格管控，打造一款高并发、零崩溃的离线审计引擎。

0x01 破题与痛点：在线爆破的穷途末路与高级库的脆弱

在实战渗透中，针对鉴权接口的传统在线爆破（Online Brute-force）早已失效。现代架构下的 WAF、API 网关限流以及动态验证码，足以将任何在线枚举行为扼杀在摇篮里。因此，获取 Token 后的“离线破解”成为了唯一可行的路径。

但在开发离线扫描器时，很多开发者习惯直接引入 PyJWT 等高级封装库来处理解析和校验。这种高度依赖上层 API 的做法在实验室环境中运行良好，一旦投入真实的复杂网络环境，就会暴露致命的工程缺陷：极差的容错率。

真实业务线返回的数据包往往充满不可预期的脏数据。一旦抓取到结构残缺、Base64Url Padding 丢失或带有 None 算法标识的畸形 JWT，高级库会在反序列化阶段直接抛出未捕获的异常（如 DecodeError）。如果不做极其繁琐的异常捕获，整个高并发扫描队列会在遇到第一个畸形 Payload 时直接崩溃中断。

为了保障高并发环境下的绝对稳定性，我们需要造一个更底层的轮子。

0x02 核心原理：HS256 离线字典碰撞的本质

抛开各种封装好的类库，JWT 的结构极其简单：Base64Url(Header) . Base64Url(Payload) . Signature。

在大多数未严格配置的业务线中，开发者常采用 HS256（HMAC-SHA256）作为签名算法。HS256 是一种对称加密方案，意味着签发 Token 和验证 Token 使用的是同一个 Secret Key。如果这个 Key 是诸如 123456 或 admin123 的弱口令，整个鉴权体系就会瞬间土崩瓦解。

HawkEye-Scanner 的核心探测逻辑即建立在此之上。我们在获取到目标 JWT 后，完全切断与服务器的交互，在本地利用 .txt 弱口令字典进行离线重组：

  1. 提取原始 Token 的 Header 和 Payload 部分。
  2. 遍历本地字典，将字典中的密码作为 Secret Key。
  3. 使用 HMAC-SHA256 算法，对 Header.Payload 重新计算签名。
  4. 将计算出的签名进行 Base64Url 编码，并与原始 Token 的第三部分（Signature）进行精确的字符串比对。

一旦比对成功，即意味着本地命中了目标服务器的 Secret Key。由于纯离线运算，这套机制规避了所有基于流量的网络防护规则。

0x03 底层重构：容错机制与字节流的精确管控

离线碰撞的逻辑虽然清晰，但要保证它在面对成千上万个格式各异的 Token时不崩溃，必须将数据清洗下沉到最底层的字节流（Bytes）级别。

处理 Base64Url 的协议变种与截断
JWT 规范为了保证 URL 传输安全，采用了 Base64Url 变种（将 + 替换为 -，/ 替换为 _），并强制去除了末尾的 = 填充符（Padding）。
然而，Python 标准库的解码器具有严格的长度校验机制。如果直接喂入截断后的数据，必定触发 binascii.Error: Incorrect padding。为了实现高可用解析，引擎在底层接管了清洗工作：通过 len(data) % 4 动态计算缺失长度，手动补齐 = 号，确保畸形数据在进入反序列化前已被标准化。

严控 String 与 Bytes 的边界转换
在进行 HMAC 碰撞时，密码学底层接口（如 hmac.new()）是极其严苛的，它只接受二进制机器字节流（Bytes），而拒绝抽象的字符串（String）。
为了避免隐式类型推导带来的运行时崩溃，工具在调度中严格把控了变量的物理形态：
在进入加密引擎前，通过 .encode('utf-8') 将所有字符串显式转换为底层字节流；在摘要生成后，再次通过 .decode('utf-8') 将二进制数据反序列化为人类可读的字符串，以进行最终的哈希比对。这种隔离设计，从根本上消灭了类型不匹配导致的程序异常。

0x04 架构演进与实测闭环

在完成了坚实的底层算力建设后，我们将目光转向了调度层的并发性能。

面对海量资产，传统的单线程同步 I/O 极其低效。HawkEye-Scanner 引入了 ThreadPoolExecutor 并发模型，将“网络请求（抓取 Token）”与“CPU 计算（离线碰撞）”在流水线中进行异步映射。通过配置 50 线程池，成功打破了网络 I/O 阻塞。

与此同时，我们在网络层引入了极端的防御性编程（Defensive Programming）。面对不同业务团队千奇百怪的 JSON 返回格式，采用链式 .get() 提取目标字段；并对整个发包与解析模块实施了异常静默（Swallow Exception）。这意味着，无论目标 IP 超时、端口拒绝还是返回了无法解析的乱码，异常线程都会被安全抛弃并释放资源，绝不将错误传递给主线程。

实测验证
在研发末期，我基于 Flask 框架搭建了本地微服务靶机。在 50 线程的高压测试下，引擎稳定完成了“自动化并发发包 -> 脏数据脱壳 -> 内存级离线碰撞”的 DAST 闭环，实现了 0% 的崩溃率，并在毫秒级内成功从字典中碰撞出漏洞靶机的弱口令。

安全工具的工程化本质，是寻找运行效率、架构鲁棒性与密码学底层原理之间的最佳平衡。本项目的核心模块已完成工程化解耦，完整源码已开源至 GitHub：HawkEye-Scanner。

黑盒测试复盘：突破隐蔽目录的 WP File Manager 漏洞利用全过程

发表于 2026-05-10 更新于 2026-05-22 分类于 实战靶场复盘

0x00 前言

在常规的自动化渗透测试中，安全工具高度依赖目标系统的默认特征库。然而在真实的攻防场景下，防御方常通过改变默认配置（如重命名高危目录）来实施特征隐藏（Security through Obscurity），从而规避扫描。

本次实战记录了一次针对 Windows 11 平台上 WordPress 靶机的黑盒测试过程。本文详细复盘了在常规扫描器漏报的情况下，如何通过调整 Fuzzing 策略发现隐蔽目录，利用元数据进行指纹识别，并最终手工构造 Multipart 表单数据包触发 WP File Manager (CVE-2020-25213) RCE 漏洞的全过程。

0x01 测试环境与拓扑架构

为了最大程度还原真实内网环境的跨平台攻防交互，本次测试采用完全本地化的物理机与虚拟机结合架构：

  • 攻击机 (Attacker)：Kali Linux (IP: 10.236.86.169)
    • 环境构建细节：未使用预装版镜像，而是采用 Debian Base ISO 进行原生编译安装。此举彻底解决了在 VMware 平台下运行渗透测试工具时偶发的鼠标指针漂移与底层显卡驱动冲突问题。同时，考虑到后期 Fuzzing 字典与抓包数据的存储需求，手动划分了 80GB 的 LVM 虚拟磁盘空间。
  • 靶机 (Victim)：Windows 11 (IP: 192.168.31.213)
    • 基础架构：采用 phpStudy 构建 Web 运行环境（Apache + MySQL + PHP）。
    • 应用框架：部署原生 WordPress CMS 系统。
    • 防御假设设定：为模拟真实的运维防御动作，靶机上安装了存在已知漏洞的插件，但管理员手动修改了该插件在 /wp-content/plugins/ 下的默认目录名称，以此来测试自动化工具的检出能力。
  • 网络拓扑：攻防双方均处于 192.168.31.0/24 及 10.236.86.0/24 路由可达的局域网环境中。

1. 常规漏扫工具的盲区

技术分析：WPScan 的插件枚举逻辑主要依赖于发送 HTTP 请求探测 /wp-content/plugins/[插件原生名称]/ 路径。一旦防御者重命名了该目录（即改变了 Plugin Slug），基于静态规则库的传统扫描器便会完全失效。

深度研判：如何排除假阳性（False Positives）干扰？
面对满屏红色的 403 和 301 状态码，如果不加甄别极易陷入“假阳性”陷阱。为确认第一轮扫描是否真正无果，我通过以下两个底层维度对海量日志进行了严格的降噪排查：

  1. 基于触发特征的 403 拦截甄别：
    日志中大量出现的 403 报错（如针对 %C0%AE%C0%AE%C0%AF 等 URL 编码变体的探测），并非意味着存在此类奇葩目录。这是因为扫描器的 Payload 携带了明显的“目录穿越（Directory Traversal）”恶意特征，在抵达业务层前，直接触发了目标服务器底层的安全规则（WAF/Apache 防护）而被强制阻断。这些响应纯属“规则拦截噪音”。

  2. 基于响应包体积（Content-Length）与手工介入的 301 甄别：
    扫描器虽捕获了大量 301 重定向（如 /admin/、/phpmyadmin/），但横向对比发现，这些响应数据包的体积高度一致。经手工介入跟进请求（Manual Verification）证实，这些路径最终均被 WordPress 底层的“目录规范化重定向（Canonical Redirect）”统一抛转至了默认的 404 报错主页（即 Soft 404 现象）。

阶段性研判结论：
第一轮扫描产生的所有异常状态码，均被证实为底层路由分发与安全策略阻断所引发的“环境噪音”，未发现任何指向真实有效资产的突破口。这反向印证了目标系统采用了极度偏门的目录命名规则，完美逃逸了常规 2W+ 级别字典的火力覆盖范围。战术必须立刻升级。

0x03 漏洞利用：指纹识别与系统接管

锁定了 /bomb/ 目录后，接下来的核心任务是确认该隐蔽组件的真实身份，并寻找可利用的攻击面。

1. 组件指纹识别 (Fingerprinting)

在纯黑盒环境下，直接对未知目录发起攻击是不明智的。依据 WordPress 插件开发的目录树规范，绝大多数合规插件会在其根目录下保留 readme.txt 或 changelog.txt 等元数据文件。

结合开源威胁情报（OSINT）及 CVE 漏洞库检索，确认该版本存在极其严重且易被利用的远程代码执行漏洞（CVE-2020-25213）。

2. 漏洞定级与载荷锻造 (Payload Crafting)

CVE-2020-25213 的成因在于，WP File Manager 底层依赖了开源的 elFinder 文件管理器组件。开发者在发布正式版时，严重疏忽地将一个用于本地测试的无鉴权接口文件（connector.minimal.php.dist）去除了 .dist 后缀。这导致任何人都可以通过该接口直接调用底层的 C 端文件操作 API。

在明确了攻击链路后，我选择在本地手工锻造一个极简的 PHP 一句话木马（WebShell），而非依赖 MSF 等重量级框架，以保障载荷的纯净度与隐蔽性：

3. 基于 elFinder API 的后门投递

漏洞利用阶段，我规避了常规的图形化上传，直接使用 curl 模拟浏览器底层行为，向存在缺陷的 connector.minimal.php 接口发送 Multipart 表单数据包。

数据包构造的底层硬核解析：

  • cmd=upload & target=l1_Lw：这是 elFinder 底层 API 的特定语法。l1_Lw 是插件工作根目录经过 Base64 变种编码后的标识符。该指令强制底层引擎将文件写入指定的高权限目录。
  • upload[]=@payload.php（致命细节）：此处必须在参数名后附加 []。因为目标服务器的 PHP 引擎及 elFinder 核心代码在处理批量文件上传时，严格要求数据类型为数组（Array）。如果不加 [] 触发数组强制转换，PHP 引擎将抛出类型不匹配异常，导致 Payload 无法落地。

  1. 彻底移除测试代码：切勿在生产环境中遗留如 connector.minimal.php.dist 类的无鉴权测试接口，发布前务必进行严格的代码审计。
  2. 及时更新组件：摒弃单纯的“重命名隐藏”策略，恢复默认的插件更新机制，并及时将 WP File Manager 升级至 6.9 及以上安全版本。
  3. 纵深防御机制 (Defense in Depth)：
    • Web 层：配置 WAF 规则，针对非法的 Multipart 文件上传及带有 target=l1_Lw 特征的请求进行拦截。
    • 系统层：严格降权 Web 服务的运行权限（如限制在受控的 www-data 或特定普通用户组），并做好网段隔离，遏制 RCE 触发后的内网跨网段横向移动风险。
